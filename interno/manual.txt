```
PÚBLICO (sin login)          ADMIN (con login)             HERRAMIENTAS
─────────────────────        ──────────────────────        ─────────────────────
index.html                   dashboard.html                sync-disponibilidad.html
cabana.html                  reservas.html                 corregir-calendar-ids.html
                             presupuestos.html             diagno.html
                             pagos.html

                             calendario.html
                             cabanas-admin.html  ← "Contenidos"
                             clientes.html
                             usuarios.html

COMPARTIDO
──────────
utils.js          → Firebase, autenticación, nav, funciones de negocio, uploader de fotos
design-system.css → tokens visuales, componentes UI

MIGRACIÓN (ejecutar una sola vez)
──────────────────────────────────
migrar-v4.html    → inicializa config/github, config/site y espacios_comunes en Firestore
```

**Principio de arquitectura:**
Todo el sistema usa exclusivamente Firestore. No hay Realtime Database en ningún flujo activo. El token de GitHub se guarda en Firestore (`config/github`) y se gestiona desde el panel admin.

# 2. BASE DE DATOS (FIRESTORE)

## Colecciones del sistema

| Colección | Acceso | Descripción |
|---|---|---|
| `cabanas` | Lectura pública | Datos de cada cabaña: nombre, fotos, capacidad, tarifas, comodidades, Calendar ID |
| `disponibilidad` | Lectura pública | Espejo de fechas ocupadas sin datos personales. El sitio público lo lee para el calendario |
| `espacios_comunes` | Lectura pública | Fotos y descripción de áreas compartidas (piscina, barbacoa, etc.) |
| `presupuestos` | Creación pública, resto admin | Formulario del sitio web crea documentos; solo admin puede leer y gestionar |
| `reservas` | Solo auth | Reservas con todos los datos del huésped |
| `pagos` | Solo auth | Ingresos registrados vinculados a reservas |
| `gastos` | Solo auth | Egresos operativos |
| `honorarios` | Admin completo, user sus propios | Pagos pendientes a colaboradores generados al finalizar tareas |
| `tareas` | Solo auth | Tareas activas (limpieza y rutinarias) |
| `tareas/{id}/sesiones` | Solo auth | Subcolección: sesiones de trabajo por colaborador |
| `historial_tareas` | Solo auth | Archivo de tareas finalizadas con horas y montos |
| `clientes` | Solo admin | Directorio de contactos |
| `usuarios` | Auth (cada uno su doc), admin todo | Roles y estado de acceso |
| `config/github` | Solo admin | Token de GitHub para subida de fotos |
| `config/site` | Solo admin | Textos editables del sitio público (título y descripción del index) |

## Estructura de documentos clave

### cabanas/{id}
```
{
  id: 1,                          // número identificador
  orden: 1,                       // orden de aparición en el sitio
  activa: true,
  nombre: { es, pt, en },
  descripcion: { es, pt, en },
  specialNote: { es, pt, en },    // nota especial opcional
  capacidad: { base: 2, max: 6 },
  camas: { dobles: 2, individuales: 1 },
  comodidades: ["WiFi", "Piscina", ...],
  fotos: [
    { url, titulo, descripcion, orden }  // formato actual v4
  ],
  tarifas: {
    precioBase: 250,
    precioExtraPersona: 50,
    precioLimpieza: 150,
    intervalos: [{ desde, hasta, precioNoche, precioExtra, descripcion }]
  },
  airbnb_url: "https://...",
  google_calendar_id: "abc@import.calendar.google.com"
}
```

### espacios_comunes/{slug}
```
{
  slug: "piscina",
  orden: 1,
  nombre: { es, pt, en },
  descripcion: { es, pt, en },    // aparece bajo cada foto en el slider
  fotos: [
    { url, titulo, descripcion }
  ]
}
```

### reservas/{id}
```
{
  caba: 1,
  estado: "confirmada",           // pendiente | confirmada | anulada | finalizada | airbnb_activa | airbnb_cancelada
  origen: "directa",              // directa | airbnb
  nombre, email, telefono,
  adultos, ninos, huespedes,
  checkIn, checkOut,              // Timestamps de Firestore
  horaLlegada, horaSalida,
  mascotas, niniosPequenos,       // "si" | "no"
  notas,
  totalBRL, costoLimpiezaBRL,
  airbnb_google_id,               // solo reservas de Airbnb
  creadoEn, creadoPor
}
```

### tareas/{id}
```
{
  nombre, descripcion,
  tipo: "limpieza" | "rutinaria" | "general",
  prioridad: "alta" | "media" | "baja",
  estado: "pendiente" | "en_curso" | "finalizada",
  fechaInicio,                    // string YYYY-MM-DD
  recurrencia: 0,                 // 0 = no recurrente, N = cada N días
  monto: 150,                     // honorario total a repartir
  activa: true,
  reservaId,                      // solo tareas de limpieza
  cabana,
  sesionesActivas: [],            // array de sesiones abiertas (snapshot para la UI)
  creadoEn, creadoPor
}
```

### config/github
```
{ token: "ghp_xxx...", actualizadoEn, actualizadoPor }
```

### config/site
```
{
  descTitle: { es, pt, en },      // título de la desc-card del index
  descBody:  { es, pt, en }       // párrafo de la desc-card del index
}
```

# 3. REGLAS DE SEGURIDAD (FIRESTORE)

```javascript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // Público — lectura sin login
    match /cabanas/{doc}          { allow read: if true;   allow write: if request.auth != null; }
    match /disponibilidad/{doc}   { allow read: if true;   allow write: if request.auth != null; }
    match /espacios_comunes/{doc} { allow read: if true;   allow write: if request.auth != null; }

    // presupuestos — creación pública (formulario web), resto requiere login
    match /presupuestos/{doc} {
      allow create: if true;
      allow read, update, delete: if request.auth != null;
    }

    // Todo lo demás requiere login
    match /{document=**} { allow read, write: if request.auth != null; }
  }
}
```

**Nota:** La separación de permisos admin/user se maneja en el código JavaScript mediante `verificarAuth()`, no en las reglas de Firestore. Esto es intencional dado el equipo pequeño y conocido.

# 4. ROLES DE USUARIO

| Rol | Acceso | Páginas |
|---|---|---|
| `admin` | Acceso completo | Todos los módulos |
| `user` | Acceso restringido | Solo `tareas.html` y su tab de honorarios en `pagos.html` |

**Campo `activo`:** Si está en `false`, el usuario no puede iniciar sesión aunque exista en Firebase Auth. Permite suspender accesos sin eliminar el historial.

**Cómo crear un usuario nuevo:**
1. `usuarios.html` → Nuevo usuario → ingresás email, nombre y rol
2. El sistema crea la cuenta en Firebase Auth y el documento en Firestore
3. Se envía email de configuración de contraseña al nuevo usuario

## index.html
**Acceso:** Sin login · Multilenguaje (ES / PT / EN)

### Propósito
Página principal del complejo. Presenta el lugar, muestra las cabañas con slider de fotos, capacidad y precio base, y exhibe los espacios comunes con un slider unificado. Permite consultar disponibilidad y generar presupuestos que se envían por WhatsApp.

### Secciones en orden visual
1. **Hero** — logo, tagline, badges de características (3 cuadras de la playa, piscina, etc.)
2. **Descripción general** — título y párrafo editable desde el panel admin (Contenidos → Configuración)
3. **Cabañas** — grilla con slider de fotos, nombre, capacidad, comodidades y precio
4. **Espacios comunes** — slider único con todas las fotos de áreas compartidas, con título y descripción por foto
5. **Contacto** — botón WhatsApp y datos de contacto

### Flujo del visitante
```
1. Llega al sitio → ve las cabañas con fotos y precio base
2. Clic en "Ver disponibilidad" de una cabaña
   → se abre modal con calendario del mes actual
   → fechas ocupadas en rojo (lee colección disponibilidad)
3. Selecciona rango de fechas
   → el sistema calcula precio automático
   → desglose: noches × precio/noche + extra personas + limpieza
4. Clic en "Consultar por WhatsApp"
   → se abre WhatsApp con mensaje precargado:
      cabaña, fechas, personas, precio estimado, idioma seleccionado
```

### Gestión de contenido (todo editable sin tocar código)
- **Fotos de cabañas** → Contenidos → Cabañas → Editar cabaña → Agregar foto
- **Fotos de espacios comunes** → Contenidos → Espacios comunes
- **Descripción general del sitio** → Contenidos → Configuración → Presentación del sitio
- **Datos de cada cabaña** (nombre, descripción, capacidad, tarifas) → Contenidos → Cabañas

### Idioma
Botones ES / PT / EN en el header. Todo el contenido cambia: textos del sitio, mensajes de WhatsApp, formato de fechas y números.

## cabana.html
**Acceso:** Sin login · Multilenguaje · URL: `cabana.html?id=1`

### Propósito
Página de detalle de una cabaña individual. Galería de fotos completa, descripción larga, lista de comodidades, calendario propio y presupuesto detallado.

### Flujo del visitante
```
1. Llega desde index.html o link directo
2. Ve galería completa con todas las fotos de la cabaña
3. Lee descripción completa y lista de comodidades
4. Consulta el calendario con fechas ocupadas de esa cabaña
5. Selecciona fechas → presupuesto detallado con desglose
6. Botón WhatsApp con todos los datos precargados
```

### Integración Firestore
- Lee `cabanas/{id}` para todos los datos
- Lee `disponibilidad` filtrado por `caba == id` para el calendario
- Lee `google_calendar_id` de la cabaña y consulta la Google Calendar API
  para mostrar también los eventos de Airbnb en el calendario

# 6. PANEL ADMINISTRATIVO

---

## utils.js — módulo central
**Sin UI propia. Todas las páginas admin lo cargan antes de su propio script.**

### Qué contiene
- Configuración de Firebase (Firestore + Auth)
- Constantes del sistema (estados, prioridades, colores)
- Configuración de GitHub (owner, repo, branch, dominio)
- Todas las funciones de negocio compartidas
- Uploader de fotos a GitHub
- Navegación centralizada

### Funciones de negocio principales

**`verificarAuth(rol)`**
Verifica sesión activa y rol del usuario. Si no cumple, redirige a `index.html`. Timeout de 15 segundos. Se llama al inicio de cada página admin.

**`renderNav(paginaActiva, rol)`**
Genera la barra de navegación según el rol. Admin ve todos los módulos; user solo Tareas y Mis cobros.

**`calcularPrecio(cabana, checkIn, checkOut, adultos, ninos)`**
Calcula precio total considerando tarifas base e intervalos de temporada por fecha. Devuelve `{ noches, subtotal, limpieza, total }`.

**`crearTareaLimpieza(reservaId, reservaData, creadoPor)`**
Crea automáticamente la tarea de limpieza al confirmar una reserva. Busca la próxima reserva en la misma cabaña para informar al personal de limpieza quién entra después.

**`sincronizarDisponibilidad(reservaId, reservaData)`**
Escribe, actualiza o elimina el espejo en `disponibilidad`. Se llama automáticamente en cada operación que cambia el estado de una reserva. Es lo que mantiene el calendario público siempre actualizado.

**`urgenciaTarea(tarea)`**
Semáforo de urgencia basado en el ciclo de recurrencia:
- Gris → fecha futura
- Verde → hoy, dentro del ciclo, o en curso (alguien trabajando)
- Amarillo → superó 1 ciclo de recurrencia
- Rojo → superó 2 ciclos, o más de 10 días

**`abrirUploaderFoto({ contexto, categoria, onConfirm })`**
Modal para subir fotos a GitHub o registrar una URL externa. Valida tamaño (máx 500 KB), muestra preview, pide título y descripción. Lee el token de `config/github` en Firestore. Sin Realtime Database.

**`iniciarTarea / pausarTarea / finalizarTarea`**
Gestión del ciclo de vida de tareas. `finalizarTarea` calcula horas proporcionales por colaborador, genera honorarios en Firestore, mueve la tarea a `historial_tareas` y la elimina o resetea según si tiene recurrencia.

## dashboard.html
**Acceso:** Admin

### Propósito
Pantalla de inicio del panel. Estado general del negocio de un vistazo y accesos rápidos a las operaciones más frecuentes.

### KPIs (se cargan al abrir)
- Reservas activas (confirmadas + pendientes)
- Pendientes de confirmación

### Accesos rápidos
Nuevo presupuesto · Finanzas · Nuevo cliente · **Contenidos** · Calendario · Tareas · Sincronizar Airbnb

### Sincronización Airbnb
Corre automáticamente al abrir el dashboard. También tiene botón manual.
```
1. Lee google_calendar_id de cada cabaña
2. Consulta Google Calendar API buscando eventos "Reserved"
3. Crea reservas nuevas en Firestore (estado: airbnb_activa)
4. Detecta cancelaciones → marca airbnb_cancelada + libera disponibilidad
5. Detecta reservas Airbnb sin google_id → muestra aviso para revisión manual
6. Muestra resultado con conteo de cambios
```

### Panel de mantenimiento (colapsable)
- Sincronizar disponibilidad
- Corregir Calendar IDs
- Diagnóstico calendarios

## reservas.html
**Acceso:** Admin

### Propósito
Gestión completa del ciclo de vida de reservas: crear, editar, confirmar, anular, finalizar.

### Flujo desde el panel
```
PENDIENTE → CONFIRMADA → FINALIZADA
    ↓             ↓
 ANULADA       ANULADA
```

### Estados
| Estado | Significado |
|---|---|
| `pendiente` | Creada pero sin confirmar (bloquea calendario) |
| `confirmada` | Confirmada con pago. Genera tarea de limpieza automáticamente |
| `anulada` | Cancelada. Libera disponibilidad |
| `finalizada` | Check-out realizado |
| `airbnb_activa` | Importada de Airbnb, activa |
| `airbnb_cancelada` | Cancelada desde Airbnb |

### Correcciones activas en v4
- Al editar una reserva confirmada, la tarea de limpieza se **elimina y regenera** desde `utils.js` para incluir siempre el bloque completo con los datos del próximo huésped
- Al confirmar una reserva, verifica si ya existe tarea de limpieza para evitar duplicados

## tareas.html
**Acceso:** Admin y User

### Propósito
Coordinación del trabajo operativo. Principal interfaz para el personal de limpieza y mantenimiento. Diseñado para uso móvil.

### Tipos de tareas
| Tipo | Origen | Recurrencia |
|---|---|---|
| Limpieza | Automático al confirmar reserva | No (se elimina al finalizar) |
| Rutinaria | Creada manualmente | Configurable en días (ej: cada 14 días) |
| General | Creada manualmente | Opcional |

### Flujo del colaborador (tareas.html en celular)
```
1. Abre tareas.html → ve sus tareas ordenadas por urgencia
2. Lee la descripción ANTES de iniciar (datos del huésped, próximo check-in, alertas)
3. Presiona "Iniciar" → registra sesión con hora de inicio en subcolección sesiones
4. Puede pausar si necesita interrumpir (el tiempo se acumula)
5. Cuando termina, presiona "Finalizar"
   → si tiene monto asignado: se calculan honorarios proporcionales por tiempo trabajado
   → la tarea se mueve a historial_tareas
   → si es recurrente: se resetea con nueva fecha
```

### Descripción de tarea de limpieza (generada automáticamente)
```
🚪 ENTRADA
   Cabaña: [nombre]
   Huésped: [nombre del próximo huésped]
   Personas: [cantidad]
   Llega: [fecha] a las [hora]hs
   Notas: [observaciones]

⚠️ ATENCIÓN: 🐾 MASCOTAS · 👶 NIÑOS PEQUEÑOS   ← solo si aplica

🔑 SALIDA (huésped anterior)
   [nombre] · [cantidad] personas
   Sale: [fecha]

💰 Monto limpieza: R$ [monto]
```

### Semáforo de urgencia (visible en la card de cada tarea)
- 🔵 Gris → fecha futura (aún no llegó)
- 🟢 Verde → hoy, dentro del primer ciclo, o alguien ya está trabajando
- 🟡 Amarillo → superó 1 ciclo de recurrencia
- 🔴 Rojo → superó 2 ciclos, o más de 10 días sin recurrencia

## cabanas-admin.html — "Contenidos"
**Acceso:** Admin

### Propósito
Gestión completa del contenido del sitio público: cabañas, espacios comunes, textos editables y configuración del sistema. Renombrado como "Contenidos" en la navegación a partir de v4.

### Tab Cabañas
Crear, editar y desactivar cabañas. Campos disponibles:
- ID numérico y orden de aparición
- Nombre y descripción en ES / PT / EN
- Capacidad (base y máxima) y camas (dobles e individuales)
- Comodidades (una por línea — no incluir capacidad ni camas para evitar duplicados en el sitio)
- Fotos con uploader integrado (archivo o URL)
- Tarifas base (precio/noche, extra/persona, limpieza)
- URL de Airbnb y Google Calendar ID

**Nota sobre comodidades:** el sitio público filtra automáticamente cualquier tag de comodidades que mencione personas, camas, dormitorios o baños, porque esos datos ya se muestran como datos estructurados. No es necesario incluirlos.

### Tab Tarifas por temporada
Períodos especiales que sobreescriben el precio base durante fechas específicas. El resto de las fechas usa el precio base de la cabaña.

### Tab Espacios comunes
Categorías de áreas compartidas (piscina, barbacoa, hamacas, etc.). Cada categoría tiene:
- Nombre en ES / PT / EN
- Descripción en ES / PT / EN (aparece bajo cada foto en el slider del index)
- Fotos con uploader (archivo o URL, máx 500 KB)
- Carpeta en GitHub (define la ruta `img/common/{carpeta}/`)

### Tab Configuración
- **Presentación del sitio:** título y párrafo de la descripción general del index, en los tres idiomas. Se guarda en `config/site`.
- **Token de GitHub:** el Personal Access Token para subir fotos al repositorio. Se guarda en `config/github` en Firestore — nunca en el código ni en pantalla. El campo siempre aparece vacío por seguridad; al abrir la pestaña solo confirma si hay un token activo. Para reemplazarlo se ingresa el nuevo y se guarda.

# 7. FLUJOS COMPLETOS DE OPERACIÓN

---

## Flujo A — Reserva directa (desde consulta hasta check-out)
```
1. Visitante consulta por WhatsApp desde el sitio público
   → mensaje precargado con cabaña, fechas, personas y precio estimado

2. Admin genera presupuesto
   presupuestos.html → Nuevo presupuesto
   → selecciona cabaña, fechas, cantidad de personas
   → el sistema calcula precio con tarifas de temporada
   → guarda presupuesto (estado: pendiente)
   → disponibilidad NO se bloquea todavía

3. Cliente acepta y paga
   presupuestos.html → Registrar pago
   → admin ingresa monto y método de pago
   → el sistema:
      • crea reserva (estado: confirmada)
      • registra pago en colección pagos
      • genera tarea de limpieza automáticamente
      • sincroniza disponibilidad (fecha queda bloqueada)
   → admin completa datos del huésped en reservas.html

4. Check-in
   → admin verifica datos en reservas.html

5. Check-out y limpieza
   → ver Flujo C — Tarea de limpieza

6. Finalizar reserva
   reservas.html → cambiar estado a "finalizada"
   → disponibilidad se libera
```

---

## Flujo B — Reserva de Airbnb (automático)
```
1. Huésped reserva en Airbnb
2. Airbnb sincroniza con Google Calendar (~1 hora de demora)

3. Admin abre el Dashboard (o presiona "Sincronizar Airbnb")
   → el sistema lee los Calendar IDs de cada cabaña
   → consulta Google Calendar API buscando eventos "Reserved"
   → para cada evento nuevo:
      • crea reserva en Firestore (estado: airbnb_activa)
      • nombre inicial: "Reserva Airbnb — completar"
      • sincroniza disponibilidad
   → resultado visible bajo el botón

4. Admin completa los datos del huésped en reservas.html
   → nombre, contacto, notas especiales

Si el huésped cancela en Airbnb:
   → evento desaparece de Google Calendar
   → en la próxima sync: reserva se marca airbnb_cancelada
   → disponibilidad se libera automáticamente
   → reserva queda visible con badge naranja para registro histórico

Caso especial — reservas sin google_id:
   → reservas Airbnb antiguas sin campo airbnb_google_id
   → el sistema las detecta y muestra aviso al sincronizar
   → no se cancelan automáticamente: requieren revisión manual en reservas.html
```

---

## Flujo C — Tarea de limpieza
```
Generación automática:
   Al confirmar una reserva → utils.js/crearTareaLimpieza() crea la tarea
   La tarea incluye:
   • Datos del huésped que SE VA (salida, hora, notas)
   • Datos del huésped que ENTRA (nombre, personas, hora llegada, alertas)
   • Monto del honorario de limpieza

El día del checkout:
   1. Colaborador abre tareas.html en su celular
   2. La tarea aparece con semáforo verde (día de hoy)
   3. Lee toda la descripción antes de empezar
      → sabe quién entra, cuándo y si hay mascotas o niños
   4. Presiona "Iniciar" → se registra hora de inicio
   5. Puede pausar si hay interrupciones (tiempo se acumula)
   6. Al terminar presiona "Finalizar"
      → se calcula tiempo trabajado
      → se genera honorario proporcional
      → tarea pasa a historial_tareas

Si la reserva se edita después de confirmar:
   → la tarea de limpieza se elimina y regenera automáticamente
   → los datos del huésped y la próxima reserva se actualizan
```

---

## Flujo D — Tarea rutinaria (mantenimiento recurrente)
```
1. Admin crea tarea manual en tareas.html
   → nombre, tipo, prioridad, fecha de inicio, recurrencia (en días), monto

2. La tarea aparece en el listado según urgencia (semáforo)

3. Colaborador la ejecuta igual que una limpieza (iniciar / pausar / finalizar)

4. Al finalizar:
   → se generan honorarios si tiene monto asignado
   → si recurrencia > 0: la tarea se resetea con nueva fecha (hoy + N días)
   → si recurrencia == 0: la tarea se elimina
```

---

## Flujo E — Pago de honorarios
```
1. Al finalizar una tarea con monto asignado
   → el sistema calcula la proporción de cada colaborador según horas trabajadas
   → crea documentos en colección honorarios (estado: pendiente)

2. Colaborador ve sus honorarios en pagos.html → tab Mis cobros

3. Admin ve todos los honorarios en pagos.html → tab Honorarios
   → selecciona los pendientes
   → registra pago: método (efectivo, transferencia, PIX), fecha, monto
   → estado cambia a "pagado"
```

# 8. HERRAMIENTAS DE MANTENIMIENTO

Accesibles desde Dashboard → Mantenimiento. Abren en pestaña nueva.
Son herramientas de uso puntual, no parte del flujo operativo diario.

---

## sync-disponibilidad.html
**Cuándo usarlo:**
- Si el calendario público muestra fechas incorrectas
- Después de operaciones masivas de importación
- Ante cualquier inconsistencia entre reservas y disponibilidad

**Qué hace:**
Lee todas las reservas activas en Firestore y reconstruye desde cero la colección `disponibilidad`. En operación normal no es necesario porque `sincronizarDisponibilidad()` en `utils.js` mantiene todo sincronizado automáticamente.

---

## corregir-calendar-ids.html
**Cuándo usarlo:**
Cuando el sistema no importa reservas de Airbnb y sospechas que el Calendar ID está mal guardado.

**Señal de problema:**
En el resultado del sync de Airbnb aparece "Calendar ID inválido" o HTTP 404.

**Qué hace:**
Detecta cabañas que tienen guardado el código `<iframe src="...">` completo en lugar del ID limpio (`abc123@import.calendar.google.com`). Extrae el ID automáticamente y lo guarda correcto en Firestore.

---

## diagno.html
**Cuándo usarlo:**
Configuración inicial de calendarios o cuando cambia el calendario de Airbnb de una cabaña.

**Qué hace:**
Muestra y permite editar el `google_calendar_id` de cada cabaña. Incluye instrucciones para encontrar el ID correcto desde el iframe de Google Calendar.

---

## migrar-v4.html
**Uso:** Una sola vez, después de subir los archivos de la versión 4.

**Pasos que ejecuta:**
1. Verifica que el token de GitHub esté en `config/github`
2. Normaliza fotos de cabañas al formato nuevo `{ url, titulo, descripcion, orden }`
3. Crea la colección `espacios_comunes` con las 6 categorías predefinidas
4. Lista reservas Airbnb sin `google_id` para revisión manual

Cada paso es idempotente: si se corre dos veces no genera duplicados.

# 10. GUÍA DE INTERVENCIÓN EN EL CÓDIGO

Esta sección es para quien asiste al administrador en modificaciones del sistema.

---

## Antes de cualquier intervención
1. Leer este manual completo
2. El sistema activo usa **solo Firestore** — ningún archivo del sistema activo carga `firebase-database.js`
3. Toda función de negocio compartida vive en `utils.js` y se exporta en `window.CVC`
4. No duplicar lógica: si algo ya existe en `utils.js`, reutilizarlo

---

## Estructura de una página admin típica
```html
<!-- 1. SDKs Firebase -->
<script src="firebase-app.js"></script>
<script src="firebase-auth.js"></script>
<script src="firebase-firestore.js"></script>

<!-- 2. Módulo compartido -->
<script src="utils.js"></script>

<!-- 3. Lógica de la página -->
<script>
(async () => {
    // Verificar auth como primer paso — siempre
    const { user, userData } = await CVC.verificarAuth('admin');
    document.getElementById('userName').textContent = userData.nombre;
    CVC.renderNav('nombre-pagina.html');

    const db = CVC.db;
    const { escapeHtml, showToast } = CVC;

    // ... lógica de la página
})();
</script>
```

---

## Cómo agregar un campo nuevo a una cabaña
1. Agregar el campo al formulario HTML en `cabanas-admin.html` (tab Cabañas)
2. En `guardarCabana()`: leer el valor del campo y agregarlo al objeto `data`
3. En `editarCabana()`: cargar el valor existente en el input correspondiente
4. En `limpiarForm()`: resetear el campo al valor por defecto
5. Si el campo debe mostrarse en el sitio público: actualizar `renderCabanas()` en `index.html`

---

## Cómo agregar una categoría nueva de espacio común
Desde el panel: Contenidos → Espacios comunes → Nueva categoría.
No requiere cambios en el código.

---

## Cómo cambiar el texto de la descripción del index
Desde el panel: Contenidos → Configuración → Presentación del sitio público.
No requiere cambios en el código.

---

## Cómo agregar un nuevo estado de reserva
1. Agregar la entrada en `ESTADOS_RESERVA` en `utils.js`
2. Agregar el CSS del badge en `design-system.css`
3. Actualizar la lógica de `sincronizarDisponibilidad()` si el nuevo estado bloquea o libera fechas
4. Actualizar los filtros en `reservas.html` y `dashboard.html`

---

## Cómo modificar el semáforo de urgencia de tareas
La función `urgenciaTarea(tarea)` en `utils.js` controla toda la lógica.
Los umbrales están documentados con comentarios en la función.
Esta función es usada por `tareas.html` y `dashboard.html`.

---

## Archivos que NO deben modificarse sin entender sus dependencias
| Archivo | Impacto si se modifica mal |
|---|---|
| `utils.js` | Afecta todas las páginas admin |
| `design-system.css` | Afecta toda la UI del panel |
| `utils.js/sincronizarDisponibilidad` | Puede desincronizar el calendario público |
| `utils.js/finalizarTarea` | Puede afectar el cálculo de honorarios |
| `utils.js/crearTareaLimpieza` | Puede perder datos que ve el personal de limpieza |
